# AI 手机桌面布局设计器 — 产品设计笔记

> 项目周期：2026.07.18 - 2026.07.21
> 开发方式：Vibe Coding（Trae AI 辅助）

---

## 一、项目背景

### 为什么做这个项目？

手机桌面是用户每天接触最多的"产品界面"，但大多数用户的桌面布局是"随缘"形成的——App 下载后随便放，久而久之越来越乱。现有手机系统（包括 ColorOS、iOS、Xiaomi HyperOS、HarmonyOS 等）只提供了"手动拖拽整理"的能力，缺乏智能化的桌面组织方案。

**这是个真实的产品问题。**

### 行业现状

- iOS：提供"App 资源库"自动分类，但桌面仍需手动整理
- ColorOS/Android：提供文件夹和小组件，但无智能推荐
- 第三方 Launcher（Nova、Niagara等）：有自定义能力，但学习成本高
- **缺失环节**：没有一款工具能根据用户习惯"自动推荐"桌面布局

---

## 二、用户画像

### 目标用户：效率型手机用户

- 年龄：20-35岁
- 特征：手机重度使用者，每天使用手机 4-8 小时
- 痛点：桌面混乱、常用 App 找不到、不知道如何科学布局
- 诉求：希望桌面高效、整洁、个性化

### 核心场景

| 场景 | 描述 | 发生频率 |
|------|------|---------|
| 场景A：新机开箱 | 新手机到手，面对空桌面无从下手 | 低频（换机时） |
| 场景B：桌面混乱优化 | 使用一段时间后桌面变乱，想整理但不知道从哪里开始 | 中频（每月1-2次） |
| 场景C：换机迁移 | 换新手机，想把旧桌面的组织逻辑迁移过来 | 低频（换机时） |
| 场景D：新增App归位 | 下载了新App，不知道放哪里合适 | 高频（每周1-3次） |

---

## 三、核心痛点（优先级排序）

### P0 — 必须解决
1. **不知道如何科学组织桌面**：用户缺乏桌面布局的知识和方法论，只能凭感觉
2. **桌面效率低下**：常用 App 找不到，不常用的占着好位置，每天浪费大量时间翻找

### P1 — 应该解决
3. **小组件不会搭配**：知道小组件有用，但不知道什么场景适合放什么组件
4. **换机迁移桌面难**：换手机后布局丢失，需要重新整理，没有智能迁移方案

### P2 — 后续迭代
5. **新增 App 智能归位**：下载新 App 后不知道放哪里，需要 AI 分析当前布局并推荐最佳位置
6. **满意度反馈闭环**：放置后持续跟踪用户满意度，不满意自动提供替代方案

## 四、产品方案

### 产品形态
Web 单页应用，直接在浏览器打开使用。

### 核心流程
```
用户输入习惯描述 → AI 分析（DeepSeek API）→ 输出可视化布局方案
```

### 输入
用户用自然语言描述自己的手机使用习惯，例如：
> "我每天刷抖音1-2小时，微信消息很多要随时回，经常用高德地图导航，
> 偶尔用美团点外卖，工作用飞书和腾讯文档，喜欢听网易云音乐"

### 输出
1. **手机桌面模拟图**（可视化）：在网页上模拟手机屏幕，展示 AI 推荐的 App 排列 + 小组件布局
2. **用户画像卡片**：AI 自动分析用户类型、核心场景、App 使用频率分层
3. **桌面布局分区卡片**：按顶部/中部/底部区域 + Dock 栏展示推荐 App 及理由
4. **设计思路说明**：2-3 句话总结核心布局逻辑

### 技术方案
- 前端框架：Vue 3（Composition API + 单文件组件，用于组件化开发和状态管理）
- 数据可视化：ECharts（用于用户习惯分析数据展示，如使用频率分布图、场景占比图等）
- 基础：HTML + CSS + JavaScript
- AI 能力：DeepSeek API（deepseek-chat 模型，温度 0.7）
- 开发方式：Trae AI 辅助 Vibe Coding

### 页面结构
- **HeroSection**：首页大标题 + 3 个渐变光球动画 + 浮动导航 Pill
- **InputPanel**：用户输入文本域 + 快速示例 + AI 分析按钮
- **LoadingOverlay**：加载动画（手机网格脉冲 + 循环提示语）
- **PhoneMock**：手机模拟框架（状态栏 + 网格区域 + Dock 栏）
- **AnalysisPanel**：用户画像 + 桌面布局分区 + 设计思路

## 五、AI Prompt 设计

### 系统 Prompt（角色设定）
```
你是一个手机桌面布局设计专家。你精通 iOS 和 Android 的桌面交互设计，
理解用户的使用习惯和效率需求。你的任务是根据用户描述的手机使用习惯，
为其设计一套最优的手机桌面布局方案。

设计原则：
1. 高频 App 放在拇指热区（屏幕下半部分）
2. 同场景 App 放在同一文件夹或相邻区域
3. 小组件应根据用户场景推荐（如通勤用户推荐地图+音乐组件）
4. 首屏不超过 24 个 App 图标
5. Dock 栏（底部固定栏）4 个 App 必须全部来自用户描述中明确提到的 App，
   不允许出现任何系统默认 App（如电话、短信、浏览器、相机等）。
   如果用户提到的 App 不足 4 个，再从高频使用场景中补充，
   但绝对不能出现系统默认 App
6. 同一个 App 只能在 grid 或 dock 中出现一次，
   不能同时在 grid 和 dock 中出现，也不能在 grid 中出现两次。
   每个 App 只出现一次。
```

### 用户 Prompt 模板（v2 迭代后）
```
请根据以下用户的手机使用习惯，设计一套最优的桌面布局方案。

用户习惯描述：
{用户输入}

请按以下格式输出（严格遵守）：

## 用户画像分析
- 用户类型：（用1-2个词概括，如"通勤社交型""学习型""效率型"）
- 核心使用场景：（列出2-3个，用逗号分隔）
- 高频App：（每天使用的，列出3-5个App名，用逗号分隔）
- 中频App：（每周使用的，列出2-3个App名，用逗号分隔）
- 低频App：（偶尔使用的，列出1-2个App名，用逗号分隔）

## 布局逻辑说明
用2-3句话总结核心设计思路，不超过80字。不要展开描述每个App的位置。

## JSON数据
输出一个JSON，格式如下。grid中的每个元素必须包含reason字段：
{
  "screen": { "rows": 6, "cols": 4 },
  "dock": [
    {"name": "App名称", "icon": "emoji", "reason": "1句话理由"}
  ],
  "grid": [
    {"name": "App名称", "icon": "emoji", "row": 0, "col": 0, "span": [1,1], "type": "app", "reason": "1句话理由", "folder": null},
    {"name": "小组件名称", "row": 0, "col": 2, "span": [2,2], "type": "widget", "reason": "1句话理由"}
  ]
}
```

### Prompt 迭代记录

| 版本 | 本次改动 | 问题 | 结果 |
|------|---------|------|------|
| v1 | 初始版 | Dock 栏返回系统默认 App（电话、短信、浏览器） | ❌ 0/4 准确率 |
| v2 | 约束 Dock 必须来自用户描述 | 仍然出现部分默认 App | ❌ 3/4 |
| v3 | 添加"绝对不允许出现系统默认 App"措辞 | Dock 4/4 正确，但 App 在 grid 和 dock 中重复出现 | ✅ 4/4 Dock |
| v4 | 新增"每个 App 只能出现一次"约束 + 前端去重函数 | 所有测试用例通过 | ✅ 4/4 去重 |

> **v2 改进说明**（2026.07.19）：删除了"桌面布局方案"文字描述段，改为前端从 JSON 数据自动生成卡片式布局。原因：AI 文字描述的位置（如"左侧"）与 JSON 中的实际 grid 坐标经常不一致，导致手机模拟图和文字说明对不上。改为从 JSON 直接渲染后，位置描述 100% 准确。同时精简了"布局逻辑说明"至 80 字以内，避免用户阅读疲劳。

## 六、UI 设计 & 动效

### 设计风格
参考 **Substance Lab**（aura.build）的设计语言：
- 配色：奶油白背景 `#F5F0EB` + 深绿文字 `#1A3320`
- 字体：无衬线字体（PingFang SC）用于标题 + 衬线斜体（Georgia）用于副标题对比
- 导航：浮动药丸形导航栏（毛玻璃效果）
- 氛围：3 个渐变光球（粉/桃/紫）作为页面背景动效

### 光球系统
- **Hero 区域**：3 个渐变光球（550px/350px/200px），边界清晰（blur: 8px + box-shadow），独立 morph 动画（7-10s 周期），50-80px 位移 + 1-3° 旋转
- **结果区域**：3 个光球定位在浏览器左右边缘，随内容滚动，不遮挡内容

### 交互细节
- 点击导航"开始设计"按钮 → 平滑滚动到输入区
- 快速体验 Chip → 一键填充示例描述
- 加载动画 → 手机网格脉冲 + 循环提示语
- **Toast 提示**：AI 生成完成后底部弹出"AI 布局已生成，下滑查看方案"，点击可滚动到结果区域，6 秒自动消失

## 七、Phase 2 规划（增强体验）

### 新增功能
1. **新增 App 智能归位**：用户输入新 App 信息 → AI 分析当前布局 → 推荐最佳放置位置
2. **备选方案轮换**：用户不满意推荐 → 提供备选方案
3. **满意度监控**：模拟 48 小时跟踪 → 弹窗询问满意度 → 不满意则重新推荐
4. **多方案对比**：同时展示 2-3 套布局方案供用户选择
5. **手动拖拽微调**：用户可在模拟手机上拖拽调整 App 位置

## 八、Phase 3 规划（结构化输入 + 真实数据）

### 核心理念
解决当前"用户不知道如何描述自己的习惯"的痛点。Phase 1 的自由文本输入虽然灵活，但对部分用户来说门槛较高——他们可能不清楚自己每天用多久、哪些算高频。Phase 3 通过结构化选项 + AI 自动判断频率，降低用户的选择困难。

### 功能 1：结构化输入引导

**用户身份选择**
- 预设选项：学生、打工人、白领
- 若选项列表中无匹配身份，用户可自行输入自定义身份
- 作用：让 AI 提前了解用户的生活场景，辅助判断 App 使用模式

**App 选择器**
- 用户从常用 App 列表中选择自己安装的 App
- 支持搜索和手动输入未列出的 App
- 作用：解决自由文本输入中用户"想不起来自己有什么 App"的问题

**频率 + 时长选择**
- 使用频率：每日使用、隔天使用、隔周使用、隔月使用
- 每日使用时长：1小时以内、2-3小时、3-4小时、4-5小时......（递进选项）
- AI 根据用户选择自动判断高频/中频/低频，无需用户自己判断
- 作用：用户只需回答"多久用一次 + 用多久"，AI 自动归类，解决选择困难

### 功能 2：导入 App 列表

**入口**
- 在"快捷体验"区域新增一个"导入 App 列表"选项

**流程**
1. 用户上传自己手机桌面的截图
2. AI 识别截图中的 App 图标和名称
3. 用户输入完布局需求后，AI 根据已导入的真实 App 列表（而非凭空推荐）进行桌面排序
4. 布局方案中的 App 100% 来自用户真实安装的应用

**价值**
- 解决"AI 推荐了用户没装的 App"的问题
- 布局方案更贴合用户真实情况
- 降低了用户逐一输入 App 名称的繁琐

### 功能 3：扫码同步（电脑 → 手机）

**核心痛点**
目前产品是 Web 端，用户在电脑上调好布局后，需要手动在手机上照着摆——这是"设计到落地"的断层。

**方案**
- 布局确认后，生成一个二维码
- 用户用手机扫描二维码
- 手机端打开一个轻量 H5 页面，展示完整布局方案
- 方案包含每个 App 的图标、位置、分页信息，用户可对照着一一摆放

**延伸可能**
- 若未来接入系统级桌面 API（如 Android Launcher 接口），可直接一键应用布局
- 短期 MVP 阶段，H5 对照页面已足够解决"电脑端操作、手机端落地"的断层

## 九、Phase 4 规划（场景化 + 个性化）

### 功能 1：换机迁移智能引导

**场景**
用户换了新手机，旧手机桌面布局全部丢失，需要重新整理。

**流程**
1. 用户在新手机上打开产品
2. 选择"换机迁移"场景
3. 系统读取旧手机的桌面使用习惯（App 列表、使用频率、布局偏好）
4. AI 预生成一套适配新手机屏幕尺寸的桌面布局
5. 用户点一下"确认应用"，无需手动输入任何需求

**价值**
- 将换机场景从"手动输入需求"降为"一键确认"
- 解决用户换机后"不知道从哪里开始整理桌面"的焦虑
- 体现产品对高价值低频场景的覆盖能力

### 功能 2：AI 布局微调选项

**核心痛点**
AI 推荐的布局虽然合理，但可能太"标准"、缺乏个性，用户觉得不够灵活。

**方案**
布局生成后，提供 3 个一键微调选项：
1. **常用 App 往边缘挪**：高频 App 从中心区移到屏幕边缘，方便单手操作，但保留原有分组逻辑
2. **换分组顺序**：保持 App 分组不变，但交换顶部/中部/底部区域的顺序
3. **紧凑模式**：缩小 App 间距，一屏容纳更多图标，适合"不喜欢翻页"的用户

**价值**
- 不改变 AI 推荐的核心逻辑，只在"最后一公里"给用户灵活选择
- 3 个选项覆盖了最常见的个性化需求，不会让用户陷入选择困难
- 用户感觉"AI 推荐的 + 我自己调的"，既有专业感又有参与感

## 十、后续产品形态迭代

### 可能性 1：内嵌为系统轻工具 / 负一屏卡片

**形态**
- 不独立存在，而是作为手机系统的内置轻工具
- 以负一屏卡片形式呈现，用户滑到负一屏即可使用

**优势**
- 完美解决"不知道用户应用列表"的痛点（系统级权限可直接读取）
- 搭配二维码双向同步：电脑网页 ↔ 手机，打通两端
- 用户无需安装额外 App，使用门槛最低
- 与 ColorOS/HyperOS/HarmonyOS 等系统深度整合，体验最流畅

**劣势**
- 仅限特定品牌手机，无法覆盖全品牌用户
- 需要与手机厂商合作，落地周期长

### 可能性 2：独立移动端 H5（轻量化）

**形态**
- 不做原生 App，只做适配手机端的响应式网页
- 全品牌手机用户均可通过浏览器访问

**优势**
- 开发成本低，一套代码适配所有手机
- 无需应用商店审核，迭代速度快
- 用户无需下载安装，扫码即用
- 与 Phase 3 的二维码同步功能天然衔接

**劣势**
- 无法读取系统级 App 列表（需用户手动选择或截图导入）
- 浏览器体验不如原生 App 流畅
- 无法在后台运行，无法实现桌面一键应用

### 可能性 3：独立安卓 App（成本最高）

**形态**
- 打包为独立 Android 应用，上架应用商店

**优势**
- 体验最流畅，可以调用系统级 API
- 可能实现"一键应用布局"到手机桌面（需 Launcher 权限）
- 品牌认知度高，独立 App 更显专业

**劣势**
- 开发成本最高，需 Android 原生开发能力
- 应用商店审核和上架流程繁琐
- 获客成本高，用户下载意愿低
- 仅覆盖 Android，iOS 需要额外开发

### 迭代路径建议

```
Phase 1-2（当前）: Web 单页应用    → 验证核心价值
Phase 3-4:         Web 应用 + 轻量 H5 → 补齐结构化输入 + 跨端体验
可能性 2:           独立移动端 H5      → 最小成本覆盖全品牌用户
可能性 1:           系统内嵌工具        → 择机与厂商合作，追求极致体验
可能性 3:           独立 App           → 长期选项，待用户量达标后考虑
```

## 十二、项目特点（面试用）

### 产品思维体现
1. **从用户痛点出发**：发现手机桌面"随缘整理"的普遍问题，而非凭空造需求
2. **产品迭代思路清晰**：首期聚焦核心功能闭环，后续规划多方案对比、拖拽微调等功能
3. **AI 产品设计能力**：Prompt 工程迭代（v1→v4），解决 AI 输出的可靠性问题
4. **UI/UX 敏感度**：参考 Substance Lab 设计语言，打造有质感的视觉体验

### 技术亮点
- Vue 3 Composition API 组件化架构
- Prompt 工程（结构化输出 + 约束条件 + 前端校验）
